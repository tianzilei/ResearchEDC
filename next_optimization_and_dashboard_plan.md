# ResearchEDC / OpenClinica Modernization：下一步优化目标与 Dashboard 建议

## 1. 背景与使用场景

本阶段优化应基于实际部署与使用规模重新定义目标。项目预期使用方式为：

- 每一个站点使用一套独立部署。
- 每个站点用户总数不超过 20 人。
- 同时在线用户不超过 10 人。
- 业务具有时间敏感性，尤其是登录、受试者管理、CRF 填写、随机化、数据审核等关键操作。

因此，系统下一步不应按照高并发互联网平台进行设计，而应按照“单站点、低并发、低尾延迟、高确定性”的业务系统进行优化。

核心目标不是横向扩容，而是：

- 降低关键操作响应时间。
- 减少页面加载链路。
- 消除慢查询和阻塞任务。
- 提高关键业务操作的稳定性。
- 避免非关键任务影响主业务流程。

---

## 2. 下一步优化目标

### 2.1 总体优化目标

下一阶段优化重点应从“功能现代化”转向“响应速度与操作确定性”。

建议将优化目标定义为：

> 系统采用单站点独立部署模式，每个站点预计用户数不超过 20 人，同时在线用户不超过 10 人。系统不以高并发扩展为主要目标，而以关键业务操作的稳定响应、低尾延迟和可预测性能为核心目标。优化将优先围绕数据库查询、接口链路、CRF 表单加载、受试者检索、随机化操作、审计日志写入和前端轻量化展开。报表导出、邮件通知、统计分析和批量数据检查等非关键任务将异步化，避免阻塞入组、随机化、CRF 保存和数据审核等时间敏感流程。

---

### 2.2 性能目标

推荐使用 P95 / P99 作为主要性能指标，而不是只关注平均响应时间。

建议初始目标如下：

| 场景 | 建议目标 |
|---|---:|
| 登录后进入 Dashboard | P95 < 1.5 秒 |
| 普通页面切换 | P95 < 800 ms |
| 受试者列表加载 | P95 < 1 秒 |
| CRF 页面打开 | P95 < 1.5 秒 |
| CRF 保存 | P95 < 1 秒 |
| 随机化操作 | P95 < 1 秒 |
| 关键业务最大可接受响应 | 不超过 2 秒 |
| 报表导出、统计分析 | 异步执行 |

对于时间敏感业务，偶发卡顿比平均速度更影响用户体验，因此需要重点关注尾延迟。

---

## 3. 推荐部署与架构方向

### 3.1 部署模式

推荐继续采用单站点独立部署：

```text
Nginx / Caddy
  ↓
Application
  ↓
PostgreSQL
  ↓
Redis，可选
```

原则：

- 每个站点独立部署。
- 应用与数据库尽量部署在同一台机器或同一局域网环境。
- PostgreSQL 作为核心数据存储。
- Redis 作为可选组件，主要用于短期缓存、任务队列或会话管理。
- 不优先引入 Kubernetes、服务网格、复杂微服务或分布式缓存。

当前使用规模下，复杂分布式架构带来的维护成本大于性能收益。

---

### 3.2 后端架构原则

后端优化重点应放在接口链路和数据库访问上：

- 所有列表接口必须分页。
- 所有筛选、排序必须在数据库层完成。
- 避免页面初始化时加载全量研究、全量受试者、全量 CRF 或全量审计日志。
- 避免 N+1 查询。
- 避免一个页面触发大量细碎接口。
- 核心业务接口保持短事务。
- 非关键逻辑从同步流程中剥离。

---

## 4. 数据库优化目标

### 4.1 优先级

数据库优化应作为下一步性能优化的第一优先级。

EDC 类系统中，响应慢通常来自：

- 受试者列表查询慢。
- CRF 表单结构加载慢。
- 权限校验重复查询。
- 审计日志写入或读取阻塞。
- 查询缺少合适索引。
- 页面一次性触发过多 SQL。
- 数据导出或统计任务阻塞业务事务。

---

### 4.2 推荐优化措施

建议优先执行以下工作：

1. 开启慢 SQL 统计。
2. 记录核心接口的 SQL 数量和总耗时。
3. 为核心业务表增加复合索引。
4. 对所有列表接口增加分页、筛选、排序限制。
5. 将审计日志查询与主业务查询拆分。
6. 将导出、报表、统计放入后台任务。
7. 控制数据库连接池大小，避免连接数过大。

推荐重点检查以下索引方向：

```text
site_id + study_id
study_id + subject_id
study_id + status
subject_id + visit_id
subject_id + form_id
form_id + status
query_status + study_id
created_at
updated_at
```

实际索引应根据慢 SQL 和真实查询模式决定，不应盲目添加过多索引。

---

### 4.3 连接池建议

由于每个站点同时在线用户不超过 10 人，连接池不应设置过大。

建议初始配置：

```yaml
application_pool:
  maximumPoolSize: 8-12
  minimumIdle: 2-4
  connectionTimeout: 3000ms
  idleTimeout: 600000ms

postgresql:
  max_connections: 50 左右
```

连接池过大可能导致数据库上下文切换增加，反而降低响应速度。

---

## 5. 关键业务路径优化

### 5.1 关键同步业务

以下操作应被视为时间敏感业务：

- 用户登录。
- 进入 Dashboard。
- 受试者筛选。
- 受试者入组。
- 随机化。
- CRF 打开。
- CRF 保存。
- 访视状态更新。
- Query 回复。
- 数据审核。

这些操作应满足：

- 短事务。
- 少查询。
- 明确索引。
- 幂等设计。
- 错误可恢复。
- 操作结果立即返回。

---

### 5.2 非关键异步任务

以下任务不应阻塞主流程：

- Excel / CSV 导出。
- PDF 生成。
- 邮件通知。
- 报表统计。
- 数据质量批量检查。
- 批量审计查询。
- 数据备份。
- 数据归档。

建议流程：

```text
用户点击任务
  ↓
创建后台任务
  ↓
立即返回任务状态
  ↓
后台执行
  ↓
完成后提供下载或提示
```

---

## 6. 前端轻量化目标

前端应服务于快速完成业务，不应追求复杂视觉效果。

推荐方向：

- 页面以中文为主。
- 不使用图片作为核心 UI 元素。
- 不使用大型 icon 库。
- 不使用复杂动画。
- 不做复杂数据大屏。
- 不在首屏加载大量图表。
- 优先使用文本、表格、表单和清晰按钮。
- 保持 daylight / night 两套轻量主题。
- 整体风格尽量接近 mono 风格。

核心原则：

> 页面越轻，关键操作越快，现场使用越稳定。

---

## 7. Dashboard 建议

## 7.1 Dashboard 定位

登录后应直接进入 Dashboard。

Dashboard 不应被设计成复杂统计首页，而应被设计成：

- 系统工作台。
- 模块入口中心。
- 待办入口。
- 最近访问入口。
- 简单系统状态提示页。

Dashboard 的核心作用是让用户快速进入业务模块，而不是在登录后加载大量统计数据。

---

## 7.2 登录流程

推荐登录流程：

```text
用户登录
  ↓
认证成功
  ↓
读取用户身份、权限、默认研究、默认站点
  ↓
直接进入 /dashboard
  ↓
显示该用户可访问的模块快捷入口
```

如果用户只有一个研究和一个站点，系统应自动进入默认上下文。

如果用户有多个研究或多个站点，Dashboard 顶部显示当前上下文，并提供切换入口：

```text
当前研究：XXX研究    当前站点：XXX中心    [切换]
```

不建议在登录后先强制进入单独的选择页，除非用户没有默认研究或默认站点。

---

## 7.3 Dashboard 页面结构

推荐结构：

```text
------------------------------------------------
ResearchEDC

当前研究：XXX研究        当前站点：XXX中心
当前用户：研究者          [切换研究/站点] [退出]
------------------------------------------------
快捷入口

[受试者管理]    [CRF填写]       [访视管理]
[随机化]        [Query管理]     [数据质控]
[问卷系统]      [数据导出]      [报表统计]
[审计日志]      [用户与权限]    [系统设置]
------------------------------------------------
待处理事项

待填写 CRF：3
待回复 Query：2
待审核数据：1
待处理账户修改申请：1
------------------------------------------------
最近访问

受试者 S001
受试者 S002
随机化记录
CRF 模板管理
------------------------------------------------
系统状态

数据库连接：正常
后台任务：正常
最近备份：2026-05-26 02:00
------------------------------------------------
```

注意：待办事项、最近访问和系统状态应异步加载，不能阻塞 Dashboard 首屏渲染。

---

## 7.4 Dashboard 加载原则

Dashboard 可以显示所有模块入口，但不能加载所有模块数据。

允许：

- 显示“受试者管理”入口。
- 显示“CRF填写”入口。
- 显示“随机化”入口。
- 显示“审计日志”入口。
- 显示“数据导出”入口。

不允许：

- 登录后立即加载所有受试者。
- 登录后立即加载所有 CRF 数据。
- 登录后立即加载所有审计日志。
- 登录后立即计算完整统计报表。
- 登录后立即生成导出数据。

推荐分层加载：

```text
第一层：模块入口，立即加载
第二层：待办数字，异步加载
第三层：系统状态，异步加载
第四层：具体模块数据，进入模块后再加载
```

## 8. Dashboard 快捷入口设计

### 8.1 推荐模块入口

Dashboard 应根据用户权限显示模块入口。建议至少包括以下模块：

| 模块 | 说明 | 推荐优先级 |
|---|---|---:|
| 受试者管理 | 入组、筛选、查看受试者状态 | 1 |
| CRF 填写 | 填写、保存、审核 CRF | 2 |
| 随机化 | 受试者随机分配与随机化记录 | 3 |
| 访视管理 | 计划访视、实际访视、访视窗口 | 4 |
| Query 管理 | 创建、回复、关闭 Query | 5 |
| 数据质控 | 缺失值、逻辑错误、异常值检查 | 6 |
| 问卷系统 | 问卷模板、分发、填写、评分 | 7 |
| 数据导出 | CSV、Excel、审计、脱敏导出 | 8 |
| 报表统计 | 入组进度、CRF 完成率、Query 统计 | 9 |
| 审计日志 | 登录、修改、随机化、权限变更日志 | 10 |
| 研究管理 | 研究配置、访视计划、CRF 绑定 | 11 |
| 站点管理 | 站点人员、权限、站点状态 | 12 |
| 用户与权限 | 用户、角色、账户修改审核 | 13 |
| 系统设置 | 系统参数、邮件、备份、部署信息 | 14 |

---

### 8.2 模块卡片样式

建议使用纯文本卡片，不使用图片和 icon。

示例：

```text
┌────────────────────┐
│ 受试者管理          │
│ 入组 / 筛选 / 状态   │
│ /subjects           │
└────────────────────┘
```

或更极简：

```text
受试者管理
入组 / 筛选 / 状态
```

不建议使用：

- SVG 插画。
- 大型 icon 库。
- 首页轮播。
- 大面积背景图。
- 动态图表。
- 大屏式统计布局。

---

## 9. Dashboard 权限控制

Dashboard 应根据用户角色和权限动态显示模块。

推荐策略：

- 用户无权限的模块默认隐藏。
- 管理员可查看全部管理入口。
- 研究者优先显示受试者、CRF、访视、Query。
- 质控员优先显示数据质控、Query、审计、报表。
- 系统管理员优先显示用户权限、系统设置、审计日志。

示例：

```text
研究者：
受试者管理、CRF填写、访视管理、Query管理、问卷系统

质控员：
数据质控、Query管理、审计日志、报表统计

系统管理员：
用户与权限、系统设置、审计日志、站点管理

研究负责人：
研究管理、受试者管理、CRF填写、随机化、数据导出、报表统计、审计日志
```

## 11. Dashboard 后端接口建议

### 11.1 首屏接口

建议提供一个轻量接口用于 Dashboard 初始化：

```http
GET /api/dashboard/bootstrap
```

返回内容只包含首屏必要信息：

```json
{
  "user": {
    "id": "u001",
    "name": "张三",
    "role": "研究者"
  },
  "context": {
    "studyId": "study001",
    "studyName": "失眠针刺研究",
    "siteId": "site001",
    "siteName": "成都中心"
  },
  "modules": [
    {
      "key": "subjects",
      "name": "受试者管理",
      "description": "入组、筛选、查看状态",
      "path": "/subjects",
      "priority": 1
    },
    {
      "key": "crf",
      "name": "CRF填写",
      "description": "填写、保存、审核CRF",
```

---

### 11.2 异步接口

待办事项单独加载：

```http
GET /api/dashboard/tasks
```

系统状态单独加载：

```http
GET /api/dashboard/status
```

最近访问单独加载：

```http
GET /api/dashboard/recent
```

这样可以避免 Dashboard 首屏被统计、审计或后台任务状态拖慢。

## 12. 验收标准

建议将以下内容作为下一阶段验收标准：

| 项目 | 验收标准 |
|---|---|
| 登录跳转 | 登录成功后直接进入 `/dashboard` |
| Dashboard 首屏 | 只加载用户信息、当前研究/站点和模块入口 |
| 权限控制 | 只显示当前用户有权限访问的模块 |
| 模块入口 | 所有核心模块均可从 Dashboard 一键进入 |
| 首屏性能 | Dashboard P95 < 1.5 秒 |
| 异步加载 | 待办、系统状态、最近访问不阻塞首屏 |
| 前端风格 | 不使用图片、复杂 icon、首页动画和统计大屏 |
| 关键业务 | 受试者、CRF、随机化入口处于第一优先级 |
| 非关键任务 | 导出、报表、批量质控异步执行 |

## 13. 下一步执行清单

### 第一阶段：性能基线

- 记录当前登录耗时。
- 记录当前 Dashboard 或首页加载耗时。
- 记录受试者列表、CRF 打开、CRF 保存、随机化接口耗时。
- 记录每个核心接口 SQL 数量。
- 建立 P50 / P95 / P99 指标。

### 第二阶段：Dashboard 改造

- 登录成功后跳转 `/dashboard`。
- 新增 Dashboard 路由。
- 新增 Dashboard bootstrap 接口。
- 根据权限返回模块列表。
- 增加研究/站点上下文显示。
- 增加最近访问和待办异步接口。

### 第三阶段：接口与数据库优化

- 检查核心接口慢 SQL。
- 增加必要复合索引。
- 所有列表接口分页。
- 拆分大接口。
- 清理 N+1 查询。
- 控制连接池大小。

### 第四阶段：前端轻量化

- 删除首页图片、图标、动画。
- 使用纯文本模块卡片。
- 简化布局。
- 减少初始化请求。
- 保持 daylight / night 两套轻量主题。

### 第五阶段：异步任务改造

- 导出任务异步化。
- 报表统计异步化。
- 批量数据质控异步化。
- 邮件通知异步化。
- 审计日志查询分页化。

---

## 14. 总结

下一步优化应明确服务于真实使用场景：单站点、小用户量、低并发、时间敏感业务。

因此，系统不应优先追求复杂分布式架构，而应优先实现：

- 登录后直接进入轻量 Dashboard。
- Dashboard 提供所有模块快捷入口。
- 首屏只加载必要数据。
- 关键业务路径保持短、快、稳定。
- 非关键任务异步执行。
- 数据库查询和接口链路优先优化。
- 前端保持极简、清晰、低资源消耗。

最终目标是让系统在小规模站点部署中具备稳定、快速、可预测的操作体验。
